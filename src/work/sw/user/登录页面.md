# Member Center SSO 登录页功能梳理

## 1. 分析范围

本次梳理聚焦以下两个主文件：

- `member-center-sso/src/main/java/com/shunwang/baseStone/sso/web/LoginAction.java`
- `member-center-sso/src/main/webapp/login/login.jsp`

为了把页面行为讲完整，另外补充追踪了两个强相关文件：

- `member-center-sso/src/main/webapp/static/login/js/login.js`
- `member-center-sso/src/main/java/com/shunwang/baseStone/sso/web/FreeLoginAction.java`

## 2. 页面整体定位

这不是一个单一的“账号密码登录页”，而是一个统一登录壳页，承接了以下能力：

1. 普通账号密码登录
2. 短信快捷登录
3. 第三方 OAuth 登录
4. 微信网页扫码登录
5. 微信公众号扫码登录
6. 微信小程序扫码登录
7. App 扫码登录
8. 网吧本地插件免登
9. 多账号 / 单账号体系下的额外绑定与授权
10. 风控极验二次验证
11. 动态口令登录分支
12. 账号注销后的撤销恢复

从实现上看：

- `login.jsp` 负责页面容器、表单、弹层、二维码区域、交互入口和前端分流。
- `login.js` 负责普通登录与短信登录的加密、校验、发送验证码等基础动作。
- `LoginAction.java` 负责站点配置初始化、登录主流程、风险控制、分支跳转、扫码回调、短信确认等后端逻辑。
- `FreeLoginAction.java` 负责网吧本地插件免登以及“免登失败后继续走扫码/授权”的补充逻辑。

## 3. 页面初始化阶段

### 3.1 静态资源与动态样式

页面会按顺序加载：

- 公共 CSS：`public.css`、`public-gray.css`
- 站点登录皮肤：优先 `loginCss.cssUrl`，否则回退 `default.css`
- 外部登录样式：`outOauthCss.cssUrl`
- 基础脚本：`jquery`、`md5.js`、`jsencrypt.js`、`qrcode.min.js`
- 登录脚本：`login.js`、`vc_plug.js`、`quick_login_min.js`
- 微信官方扫码脚本：`wxLogin.js`
- 条件脚本：`sw-pv-report-js.min.js`（仅 `needReportData` 为真时加载）

页面 `body` 的 `class` 由后端 `bodyClassName` 注入，可用于业务侧自定义极验或皮肤样式。

### 3.2 页面级全局变量

JSP 向前端注入了若干关键变量：

- `SSO_SERVER`
- `interfaceServer`
- `identityServer`
- `validPhoneRex`
- `siteId`
- `callbackUrl`
- `loginType`
- `loginMode`
- `needCheckCode`
- `needSmsCheckCode`
- `isFreeLoginFlag`
- `dllSwitch`
- `headAuthType`
- `smsSendExpTime`
- `checkTimeInterval`
- `unAuthAgreementFlag`

这些变量直接决定页面初始显示、按钮行为、二维码轮询、短信倒计时、协议勾选逻辑和扫码模式。

### 3.3 后端配置装配

后端 `initLoginElement(siteId)` 会按站点注入以下配置能力：

- 普通登录是否启用极验
- 短信登录是否启用极验
- 是否启用动态口令
- 是否启用短信快捷登录
- 是否启用立即注册
- 是否显示忘记密码
- 是否显示 App 扫码入口
- 是否启用网吧免登
- 是否启用数据上报
- 二维码轮询时间间隔
- 单账号体系开关
- 未授权协议开关

## 6. 普通账号密码登录

### 6.1 前端校验逻辑

点击 `#loginButton_YHJ` 后，前端会做：

1. 校验用户名是否为空
2. 如果 `authFreeType=2` 且未勾选 `agreementCheckbox2`，弹出授权说明弹层
3. 若当前还未强制开启极验，则先调 `/loginPreCheck.do`
4. `/loginPreCheck.do` 返回 `code=1` 时，不直接提交，而是先拉起极验

### 6.2 前端密码处理

密码不会明文提交：

1. 先读取密码框内容
2. 前端判断是否弱密码，并把结果写入 `weakPwdState`
3. 对密码做 `MD5`
4. 用前端内置的 RSA 公钥再次加密
5. 写入隐藏域 `password`
6. 清空明文密码输入框

这意味着传到后端的 `password` 已经是“前端 MD5 + RSA”的结果。

### 6.3 后端 `execute()` 主流程

`LoginAction.execute()` 是普通登录核心入口，主要顺序如下：

1. 写入 `P3P` 响应头
2. 校验站点合法性 `checkSite()`
3. 初始化站点登录配置 `initLoginElement(site_id)`
4. 校验用户名 / 密码非空
5. 校验 IP 是否被限制
6. 初始化消息和扩展参数 `initParams()`
7. 如需极验则先校验极验结果
8. 根据用户名查询账号 `UserRegisterKeyUtil.getMember(userName)`
9. 调风控服务 `RiskService.checkRisk(...)`
10. 校验用户是否存在、是否禁用
11. 根据密码错误次数判断是否必须走极验
12. 后端用私钥对 `password` 解密
13. 如未标记 `md5`，则再补一次 MD5
14. 执行真正的账号密码登录 `UserLoginContext.login(...)`
15. 根据结果决定后续分支

### 6.4 登录成功后的分支

普通登录成功不一定直接 `SUCCESS`，可能继续分流到：

#### A. 弱密码拦截

- 若站点命中“弱密码强制跳转”配置，且当前密码被判定为弱密码
- 后端会创建登录缓存，但不直接完成业务跳转
- 页面提示用户先去重置密码

#### B. 动态口令页

- 如果账号绑定了动态口令能力，且站点开启动态口令
- 后端会先发动态短信，再跳到 `dynamicPwd` 页面要求输入二次验证码

#### C. 单账号绑定页

- 如果站点是单账号体系
- 当前用户尚未完成手机号或主账号绑定
- 则跳到 `singleAccountBind` 或 `singleAccountBindForH5`

#### D. 风控二次验证页

- 如果风险服务返回“需额外验证”
- 会跳到 `geetestValided` 结果页，再由 `/geetestValid.do` 完成二次确认

#### E. 注销账户撤销页

- 如果后端返回 `C_1068`
- 表示账号已注销
- 页面会进入“可撤销注销”分支，而不是普通错误页

#### F. 异常账号安全页

- 若用户被标记为高风险 / 不安全账号
- 后端会跳到 `unsafe_success`

### 6.5 登录成功后生成的登录凭证

后端 `createLoginCache(member)` 会生成三类关键票据：

- `ticket`
- `tockenId`
- `clientTicket`

这些票据会被后续回调、扫码跳转、多端登录和客户端场景重复使用。

## 7. 短信快捷登录

### 7.1 入口与展示

- 只有后端配置 `needSmsQuickLogin=true` 时，普通登录页才显示“短信快捷登录”入口。
- 页面切换由 `.login-type-link > a` 控制，可在普通登录和短信登录之间来回切换。

### 7.2 发送验证码流程

发送验证码按钮 `#btnSendSMS` 的行为如下：

1. 校验手机号格式
2. 若短信极验开关打开，则必须先通过极验
3. 前端用 RSA 加密手机号
4. 携带 `site_id`、极验结果、`reportData` 调 `/sendSmsActiveNo.do`

后端 `sendSmsActiveNo()` 的关键点：

- 1 分钟发送频控
- 可选极验校验
- 解密手机号
- 调接口服务发送短信
- 若发送成功，返回 `isReg` 标记当前手机号是否已注册
- 同时初始化一分钟倒计时

### 7.3 协议自动勾选逻辑

短信发送成功后，如果返回 `isReg=true`，前端会自动把协议勾上。这说明：

- 短信登录不是纯技术流程，还带了协议合规逻辑
- 已注册手机号与未注册手机号，协议默认状态可能不同

### 7.4 短信确认登录流程

点击短信登录提交时：

1. 必须勾选协议
2. 校验手机号格式
3. 校验短信验证码是 6 位数字
4. 用 RSA 分别加密手机号和短信验证码
5. 提交到 `/outMobileSmsConfirm.do`

后端 `outMobileSmsConfirm()` 会：

- 初始化登录元素
- 做防刷锁 `ANTI_ADDITION_LOGIN_<number>`
- 解密手机号与验证码
- 调用接口服务做短信登录确认
- 成功后创建登录票据、写 Cookie、设置 `loginMode=smsLoginType`
- 失败则回到短信登录页并回显错误

## 8. 外部账号 OAuth 登录

### 8.1 入口来源

外部登录图标由 `outOauthDirs` 渲染生成，支持在普通登录区和短信登录区下方显示。

### 8.2 移动端过滤逻辑

`initCss()` 里会根据 `User-Agent` 动态过滤微信相关入口：

- 手机端微信外部：移除某些微信入口
- 手机端微信内部：移除另一些微信入口

这说明不同容器内展示的“微信登录”并不总是同一条链路。

### 8.3 外部登录跳转方式

`goToOtherSite(siteId)` 是总分发器：

- `1008`：公众号扫码登录
- `1013`：微信小程序扫码登录
- 其它外部站点：走 `/goOutOauth.do`

当站点是单账号体系时，还可能使用弹窗方式打开外部认证页。

## 9. 各类扫码登录

### 9.1 站点 ID 对照

| ID | 含义 |
| --- | --- |
| `1005` | 微信网页扫码 / 开放平台登录 |
| `1008` | 微信公众号扫码 |
| `1012` | App 扫码登录 |
| `1013` | 微信小程序扫码 |

### 9.2 微信网页扫码登录 `1005`

前端 `doWxChatLogin()`：

1. 序列化 `#login_form_`
2. 请求 `/getOutSiteOauth.do`
3. 后端返回 `appid`、`scope`、`redirect_uri`、`state`、`style`、`href`
4. 前端调用官方 `WxLogin(...)` 渲染二维码

这是标准的微信官方网页扫码模式。

### 9.3 微信公众号扫码登录 `1008`

前端 `doWxSubLogin()`：

1. 请求 `/getOutSiteOauth.do`
2. 取回 `qrcodeUrl`
3. 用本地 `QRCode` 插件生成二维码
4. 保存后端返回的 `key`
5. 按 `expireSeconds` 开始轮询 `/checkWxOpenLogin.do`

这个模式主要用于“公众号关注 / 扫码授权 / 扫码安全验证”场景。

### 9.4 微信小程序扫码登录 `1013`

前端 `doWxMiniLogin()`：

1. 请求 `/getOutSiteOauth.do`
2. 直接把返回的 `qrcodeUrl` 填到图片地址上
3. 记录 `key`
4. 开始轮询 `/checkWxOpenLogin.do`

### 9.5 App 扫码登录 `1012`

点击 `.app_qr_code` 或外部入口时，会：

1. 把 `loginSiteId` 设置成 `1012`
2. 调 `doAppLogin()`
3. 请求 `/getOutSiteOauth.do`
4. 用 `QRCode` 生成 App 二维码
5. 开始轮询 `/checkWxOpenLogin.do`

### 9.6 所有扫码模式共享的轮询逻辑

`checkWxSubLogin(wxsub_sec, classId)` 是核心轮询函数，状态分支如下：

- `init`：二维码刚生成，继续轮询
- `preLogin`：用户已扫码但还未确认，前端显示“已扫码待确认”
- `loginSuccess`：登录成功，跳 `/loginCallback.do`
- `singleBind`：单账号绑定成功，也跳 `/loginCallback.do`
- `userCancel`：账号已注销，切到撤销注销提示层
- `error`：显示错误文案
- 超时：显示二维码失效，提示刷新

### 9.7 扫码回调处理

后端 `loginCallback()`：

- 若带 `singleBindSign`，进入单账号绑定页
- 若带 `ticket + tockenId`，返回正常成功页

后端 `autoLoginCallback()`：

- 主要承接免登链路回调
- 会检查票据是否还有效
- 还会做多账号绑定、模板消息发送、上报等处理

## 10. 网吧本地插件免登

这是整个页面最业务化的模块之一。

### 10.1 启动条件

只有 `isFreeLoginFlag=true` 时，页面加载后才会尝试接入本地插件。

### 10.2 本地插件读取流程

前端会通过 JSONP 调：

- `https://plugin.kedou.com:9199/getMemberInfo.htm`

读取本地客户端登录信息，得到：

- 本地登录 key
- 会员信息 JSON
- `ssoAuth`
- `memberName`
- `memberId`
- 昵称、吧台信息等扩展字段

### 10.3 头像授权快速登录

如果本地信息满足头像授权条件，并且 `headAuthType` 开关允许：

- 页面会展示 `.sso-authorize-login`
- 把头像、昵称渲染出来
- 用户点击头像后才继续真正登录

这本质上是“本地已知身份 + 前端再确认一次”的登录体验。

### 10.4 直接免登

前端从本地插件检查出 `freeTicket`、`memberId` 后，有两种路径：

#### 非 Ajax 方式

- 直接把 `freeTicket` 和 `memberId` 写回表单
- 提交到 `/freeLogin.do`

#### Ajax 方式

- 请求 `/freeLogin.do`
- 后端返回不同 `qrType`
- 前端据此决定下一步展示

### 10.5 `FreeLoginAction` 返回值语义

| 返回字段 | 含义 |
| --- | --- |
| `qrType=none` | 当前不能继续免登 |
| `qrType=dll` | 已可直接走本地免登或头像授权链路 |
| `qrType=1008` | 需要继续走公众号扫码 |
| `qrType=1013` | 需要继续走小程序扫码 |
| `dataBind=true` | 多账号场景下需要引导用户做授权绑定 |
| `needHeadAuth=true` | 需要额外头像确认或网吧授权确认 |

### 10.6 `businessScene` 的作用

当后端无法直接完成免登，而需要继续走“扫码绑定 / 扫码授权 / 安全确认”时，会生成 `businessScene`：

- 它是一个缓存场景 key
- 会把 `memberName`、`businessType`、`guestType` 等上下文写进 Redis
- 前端后续把这个值再回传给扫码接口和轮询接口

也就是说，`businessScene` 是“插件本地信息”和“扫码授权链路”之间的桥梁。

## 11. 多账号免登授权

### 11.1 页面表现

普通登录区有一块默认隐藏的 `.multiple-agreement`，包含：

- 一个勾选框 `agreementCheckbox2`
- 一条免登协议链接

只有在某些免登 / 多账号返回场景下，这块区域才会显式展示。

### 11.2 展示时机

当 `FreeLoginAction` 返回：

- `qrType=none`
- `dataBind=true`

前端会：

- 把 `idCardName` 写入隐藏域
- 把 `authFreeType` 设置为 `2`
- 显示 `.multiple-agreement`

这表示“当前无法自动免登，但可以引导用户在普通登录成功时同步完成授权绑定”。

### 11.3 登录时的处理

普通登录前，如果 `authFreeType=2` 且用户未勾选协议：

- 页面不会直接提交
- 而是弹出 `.protocol-dialog` 进行说明

用户可选择：

- “授权绑定并登录”
- “不绑定，继续登录”

后者会把 `authFreeType` 改成 `4` 再继续。

### 11.4 后端处理

普通登录成功后，如果：

- `authFreeType` 不是普通登录
- `idCardName` 有值

后端会根据授权类型决定：

- 是否保存主从账号关系
- 是否调用 `multipleAccountBindService.bindAccountIfNecessary(...)`
- 是否记录免登授权埋点

## 12. 单账号绑定场景

当站点属于单账号体系时，登录过程中可能额外进入绑定流程。

### 12.1 触发场景

- 普通登录成功后发现当前账号尚未完成绑定
- 动态口令登录成功后发现尚未绑定
- 扫码登录回调带了 `singleBindSign`
- 免登流程中发现单账号信息缺失

### 12.2 页面结果

后端会返回：

- `singleAccountBind`
- `singleAccountBindForH5`
- `singleAccountBindNoJumpForH5`

### 12.3 相关接口

- `/sendSingleBindActiveNo.do`：发送绑定验证码
- `/comfirmSingleBindActionNo.do`：确认验证码并完成绑定

### 12.4 相关细节

- 单账号绑定也支持极验开关 `singleGeetestSwitch`
- 也有 1 分钟短信倒计时
- 绑定成功后同样会创建登录票据、写 Cookie、继续回调

## 13. 动态口令分支

如果账号开启了动态口令能力：

1. 用户先完成账号密码校验
2. 后端发送动态短信验证码
3. 返回 `dynamicPwd` 页面
4. 用户输入动态验证码后调用 `dynamicPwdLogin()`

`dynamicPwdLogin()` 会：

- 做参数检查
- 对用户维度做一分钟限频锁
- 调接口验证动态口令
- 成功后创建登录票据
- 继续进入单账号 / 安全页 / 正常回调分支

## 14. 注销账号撤销恢复

### 14.1 触发方式

两条链路可能触发：

1. 普通登录或短信登录时后端判断账号已注销
2. 扫码轮询过程中返回 `userCancel`

### 14.2 页面表现

前端会展示 `.weChat-repeal-tip-content`，其中包含隐藏表单：

- `optToken`
- `site_id`
- `memberName`

### 14.3 撤销动作

点击“撤销注销”时请求 `/undoCancel.do`。

后端会：

1. 校验参数不为空
2. 用 `AES(memberName + site_id)` 算出 token
3. 校验传入 `optToken`
4. 把账号状态恢复为正常
5. 异步发送撤销通知

## 15. 风控与极验

### 15.1 普通登录极验

普通登录有两层极验触发方式：

- 页面初始化就配置为需要极验
- 密码错误次数超过阈值后强制要求极验

### 15.2 短信登录极验

短信登录是否要极验，由：

- 总极验开关
- 站点短信极验配置

共同决定。

### 15.3 风控二次验证页

如果风险服务返回“不通过但可继续验证”：

- 后端会把当前登录生成的票据写入 Redis
- 生成一个短期 `optToken`
- 返回 `geetestValided` 页面

后续调用 `/geetestValid.do`：

- 极验通过则取回之前缓存的 `ticket/tokenId/clientTicket`
- 再返回可用的 `fmtCallback`

## 16. 设备 / 环境数据上报

当 `needReportData=true` 时：

- 页面会加载 `sw-pv-report-js.min.js`
- 周期性读取机器 / 网吧环境数据
- 自动填入所有 `reportData` 隐藏字段

这些数据会被：

- 普通登录
- 短信验证码发送
- 第三方登录
- 免登链路

统一带回后端，用于埋点、风控或业务分析。

另外，`reportDllData.do` 还会把插件本地识别出的 `memberName` 写回扫码业务场景缓存，供后续绑定和授权流程使用。

## 17. 皮肤、协议与文案配置

### 17.1 皮肤

后端 `initCss()` 会装配：

- 登录页 CSS
- H5 CSS
- 外部 OAuth 区域 CSS
- 用户协议 URL
- 隐私协议 URL

若站点没有单独配置，会回退到 Passport 默认配置。

### 17.2 协议展示

页面中至少有三类协议区域：

1. 短信登录协议勾选
2. 多账号免登授权协议
3. 扫码授权页中的用户协议 / 隐私协议 / 网吧免登协议

### 17.3 默认勾选逻辑

用户协议是否默认勾选，受：

- `userAgreementCss.state`
- 通用开关 `AGREEMENT_AUTO_CHECK_SWITCH`
- 白名单站点配置

共同影响。

## 18. 回调与跳转控制

### 18.1 `getFmtCallback()`

后端会把：

- `ticketId`
- `tockenId`
- `loginMode`

拼进 `callbackUrl`。

如果 `needCode=1`，还会额外去下游接口换取业务 `code` 再一起带回。

### 18.2 `tgt` 的作用

前端 `jumpUrl(json)` 会根据 `tgt` 决定：

- `self`：当前窗口跳转
- 其它值：顶层窗口跳转

这主要用于嵌入式 SSO 场景，控制是否覆盖父窗口。

## 19. 安全与防刷细节

这个页面在安全与防刷上做了较多处理：

1. 前端密码 `MD5 + RSA` 双层处理
2. 前端会清空明文密码框
3. 后端再做 RSA 解密
4. 普通登录按密码错误次数触发极验
5. 短信发送按站点做 1 分钟限流
6. 动态口令验证有一分钟计数锁
7. 短信快捷登录确认有防刷锁
8. 撤销注销使用加密 token 校验
9. 登录成功后统一写 Cookie 和安全标识
10. 支持 IP 限制

## 21. 功能总结

从产品和技术两侧看，这个页面可以概括成一句话：
