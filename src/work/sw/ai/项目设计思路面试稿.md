## 一句话介绍项目

`网维小智` 是一个面向网吧运维场景的智能助手系统，基于 `FastAPI + LangGraph + Vue` 构建，核心目标不是做一个“会聊天”的 AI，而是做一个**能理解业务意图、能调用真实运维能力、能中断确认、能持续追溯上下文**的业务型 Agent。

你可以直接这么说：

> 这个项目本质上不是传统问答机器人，而是一个面向网吧运维场景的 Agent 平台。它把大模型、业务工具、会话状态、人工确认和前端流式交互串成了一条完整闭环，让 AI 不只是“回答问题”，而是真的“参与业务动作”。

---

## 项目背景：为什么要做这个系统

传统网吧运维有几个痛点：

1. **信息分散**：游戏资源、网吧信息、用户信息、硬件资产都在不同系统里。
2. **查询链路长**：一个简单问题，比如“哪些店没有装某个游戏”，往往要点很多后台页面。
3. **动作风险高**：下载、更新这类操作一旦误触，影响的是真实门店和真实机器。
4. **上下文割裂**：传统系统一问一答还行，但多轮追问体验很差。

所以这个项目的设计目标非常明确：

- 让用户用自然语言完成复杂查询；
- 让系统自动把问题映射成工具调用；
- 对高风险动作加入人工确认；
- 对整个过程做持久化，支持恢复、回看、重新生成；
- 用流式交互把 Agent 的“思考过程”和“执行结果”透明展示给前端。

这就是这个项目最有价值的一点：**它不是把 AI 硬塞进业务，而是把业务流程 Agent 化。**

---

## 我对这个项目的整体定义

我会把它定义成四层：

### 1）交互层

- 前端用 `Vue` 做聊天式工作台；
- 后端用 `FastAPI` 提供统一 API；
- 前后端之间通过 `SSE` 做流式事件传输。

### 2）Agent 编排层

- 基于 `LangGraph` 构建状态图；
- 把一次会话拆成“意图识别 → 模型生成 → 工具调用 → 人工确认 → 标题生成”等节点；
- 通过图编排替代大模型单轮黑盒调用。

### 3）业务工具层

- 把网吧、资源、硬件相关能力封装成标准工具；
- 大模型只负责理解和路由，不直接碰底层业务细节；
- 工具输出同时支持 `content + artifact`，方便前端结构化展示。

### 4）状态与基础设施层

- 用 `MySQL` 做 LangGraph checkpoint 持久化；
- 用 `Redis Pub/Sub` 做跨请求/跨实例的停止生成广播；
- 用连接池、保活、统一日志和配置加载保障稳定性。

面试里你可以总结成一句：

> 我把这个系统设计成“前端工作台 + 后端编排引擎 + 业务工具中台 + 状态基础设施”四层结构，保证它既能快速交互，也能真正落业务。

## 核心架构亮点

## 1. 不是单 Agent 问答，而是“图驱动 Agent”

这个项目最值得讲的点，就是没有直接写一个 `prompt + llm.invoke()`，而是使用 `LangGraph` 来建图。

当前图里至少包含这些节点：

- `intent_analysis`：先判断用户意图；
- `model`：主模型生成回复或决定调用工具；
- `human_review_node`：对高风险工具做人机确认；
- `tools`：执行工具；
- `generate_title`：会话结束后自动生成标题。

这个设计的价值是：

- **流程可控**：不是完全依赖模型自由发挥；
- **状态可追踪**：每个节点都能查；
- **行为可扩展**：后续加新节点很自然；
- **更适合业务场景**：业务型 Agent 本质上需要流程，不只是文本生成。

面试表达：

> 我没有把大模型当成万能黑盒，而是把它放到一个图编排体系里，让模型负责理解和决策，让系统负责状态、流程和约束。这样做的好处是，系统从“会聊天”升级成“可执行的业务 Agent”。

## 2. 用“意图识别 + 工具裁剪”控制成本和准确率

项目不是每次都把所有工具直接扔给模型，而是先做一轮意图分析，再根据意图选择工具集。

比如：

- 资源相关问题：使用 `game_tools + common_tools`；
- 硬件相关问题：使用 `hardware_tools + common_tools`；
- 日常闲聊：只给少量通用时间工具；
- 不确定场景：再回退到全量工具。

这个设计非常适合面试讲：

- **减少提示词噪声**；
- **降低工具误调用概率**；
- **控制模型决策复杂度**；
- **本质上是在做面向业务场景的工具路由。**

而且项目里还做了两层意图识别：

1. 先走关键词的轻量级快速判断；
2. 命不中再走结构化的 LLM 意图识别。

这就是一个很典型的工程优化：

> 不是所有问题都值得走一遍完整大模型分类，能用轻规则兜住高频场景，就先用规则提速；复杂场景再交给模型做精判。

## 3. 把危险动作从“自动执行”改成“可中断审批”

这是这个项目特别像真实业务系统的一点。

例如资源下载/更新工具，不是模型一决定就直接执行，而是先进入 `human_review` 节点，弹出中断信息，等用户确认后再继续。

这套设计的意义很大：

- 避免模型误调用造成真实业务损失；
- 让 Agent 从“建议系统”升级成“可控执行系统”；
- 很符合企业对 AI 落地的核心诉求：**能干活，但必须可控。**

这个点在面试里非常加分。

你可以这么说：

> 我认为业务 Agent 最重要的不是“能不能调工具”，而是“调错了怎么办”。所以我把高风险工具前置成中断审批流，相当于给 Agent 加了一层可治理能力。这个设计比单纯追求自动化更适合真实生产环境。

## 4. 用 MySQL Checkpoint 做会话持久化，不丢上下文

项目没有把聊天记录简单丢到前端或者缓存里，而是用 `LangGraph + AIOMySQLSaver` 做状态级持久化。

这带来几个能力：

- 支持多轮会话连续追踪；
- 支持查询历史消息；
- 支持从中断点继续恢复；
- 支持“重新生成”时基于最近 checkpoint 回放；
- 支持删除整个 session 对应的 thread 状态。

这个设计比“存一堆 message 文本”高级很多，因为它存的是**图执行状态**，不只是聊天内容。

可讲话术：

> 我们持久化的不是简单聊天记录，而是 Agent 执行状态。这意味着系统不仅知道“聊过什么”，还知道“执行到哪一步”“是否被打断”“能不能恢复”，这才是真正可运营的 Agent 系统。

## 5. 用 Redis Pub/Sub 做停止生成，解决异步流式控制问题

流式生成最难的一点，不是输出，而是**停下来**。

这个项目的做法是：

- 为每个 `thread_id/session_id` 维护停止事件；
- 前端发 `/stop` 请求；
- 后端把停止消息发布到 Redis `stop_generation` 频道；
- 消费侧收到消息后，设置对应线程的 stop event；
- Agent 在意图分析和模型流式生成阶段都会检查 stop 标记。

这个设计的亮点是：

- 不把停止逻辑绑死在单进程内；
- 有利于多实例部署；
- 把“控制信号”和“业务请求”解耦。

这是非常工程化的做法，绝对值得讲。

一句话版本：

> 我们不是在当前请求里粗暴打断，而是通过 Redis 做控制信号广播，把停止生成做成了一个独立的异步控制通道，这样系统更容易扩展成多实例。

## 6. 前端不是只显示答案，而是显示 Agent 执行过程

项目前端不只是一个聊天框，它能展示：

- 会话列表和标题；
- 流式消息内容；
- 工具开始执行的信息；
- 工具结果；
- 中断审批信息；
- 停止生成与重新生成状态。

而后端 SSE 也不是只推 `message`，而是推一套事件协议：

- `checkpoint`
- `message`
- `tool_start`
- `tool_result`
- `interrupt`
- `title`
- `stop`
- `end`
- `error`

这说明我在设计时不是把前端当成“输出容器”，而是把它当成 **Agent 可视化工作台**。

这个角度也很好吹：

> 很多 Agent 项目后端做得很重，但前端仍然是黑盒。我这里专门把 Agent 的关键执行节点事件化，让用户感知到系统在“理解、调用、确认、返回”的全过程，这对体验和信任感都很重要。

## 端到端链路怎么讲

面试时建议你按一条请求链路讲，最清晰。

### 场景例子

用户说：**“帮我看看哪些网吧没有安装某个游戏，如果没装就准备下发。”**

### 系统处理流程

1. 前端调用 `/chat_stream`，把 `message + userId + brandId + sessionId` 传给后端。
2. 后端根据 `sessionId` 绑定会话线程，没有就创建新的 `thread_id`。
3. Agent 先进入 `intent_analysis` 节点，判断这是资源类问题。
4. 系统只给资源相关工具，缩小模型决策空间。
5. 主模型判断需要调用资源查询工具。
6. 工具返回结构化结果，后端通过 SSE 流式推给前端。
7. 如果模型进一步决定执行下载/更新，则不会直接落地，而是进入人工确认节点。
8. 前端展示中断问题，用户点“继续”或“拒绝”。
9. 若继续，则 Agent 从中断点恢复执行；若拒绝，则生成工具拒绝结果并回到模型节点。
10. 会话结束后自动生成标题，并把整个状态持久化到 MySQL checkpoint。

这时候你可以升华一句：

> 这套链路的本质，是把用户的一句自然语言，拆解成了“理解 → 决策 → 执行 → 审批 → 持久化 → 可视化反馈”的标准业务闭环。

## 技术设计为什么这么选

## FastAPI

我选 `FastAPI` 的原因：

- 异步能力好，适合流式输出；
- 接口定义清晰，和 Pydantic 模型配合自然；
- 做 SSE、生命周期管理、异常处理都比较顺手。

## LangGraph

我选 `LangGraph` 而不是简单链式调用，是因为：

- 业务流程有状态；
- 会中断；
- 会恢复；
- 会走条件分支；
- 需要 checkpoint。

这种场景天然适合图，而不是单链。

## MySQL

我用 MySQL 持久化 checkpoint，而不是只用内存或文件，是因为：

- 状态需要可靠保存；
- 会话需要跨进程可读；
- 历史和重生成都依赖稳定存储；
- 业务环境通常本来就有 MySQL，落地成本低。

## Redis

Redis 主要不是拿来做缓存，而是做：

- 停止生成消息通道；
- 轻量级异步控制总线；
- 多实例之间的状态控制桥梁。

这个定位非常关键，面试里可以强调你不是“为了用 Redis 而用 Redis”。

## 多模型拆分

项目里把模型拆成了三类：

- 主 Agent 模型；
- 意图识别模型；
- 标题生成模型。

这是一个典型的“职责分离”设计：

- 意图识别更偏稳定、低温度；
- 标题生成更偏轻量、便宜、快速；
- 主 Agent 模型承担复杂推理和工具决策。

你可以这么讲：

> 我没有让一个大模型承担所有角色，而是把高频分类、主对话决策、轻量标题生成拆开，这样能更好地平衡效果、成本和响应速度。

## 你可以重点讲的 8 个技术亮点

1. **LangGraph 状态图编排**：不是单轮调用，而是多节点工作流。
2. **意图识别前置**：先分类，再裁剪工具集合。

7. **SSE 事件协议化**：前端看到的不是文本流，而是 Agent 运行流。
8. **多模型职责分离**：主模型、意图模型、标题模型各司其职。

## 面试官可能会追问的问题

## Q1：为什么不用普通聊天机器人方案？

答：

> 因为业务问题不是一轮文本生成能解决的。这个场景有工具调用、状态分支、中断审批、恢复执行等需求，所以必须上图编排，而不是简单聊天链路。

## Q2：为什么要做意图识别，不让模型直接选工具？

答：

> 直接全量给工具虽然省事，但会增加提示词噪声和误调用概率。先做意图识别，本质上是在缩小模型决策空间，既提升准确率，也降低推理成本。

## Q3：为什么高风险操作要人工确认？

答：

> 因为业务 Agent 最大的问题不是“不会做”，而是“做错了代价大”。下载或更新真实影响线下门店，所以必须把自动化建立在可控基础上。

## Q4：为什么把 checkpoint 放 MySQL？

答：

> 因为我要的不是临时上下文，而是稳定持久的执行状态。MySQL 方案能支持恢复、重生成、历史查询，也更符合企业现有基础设施。

## Q5：Redis 在这里解决了什么问题？

答：

> Redis 主要解决异步控制问题，尤其是停止生成。它不是简单缓存，而是系统里的控制总线，让停止信号可以跨请求、跨实例传播。

## Q6：这个项目最难的点是什么？

答：

> 我觉得最难的不是调模型，而是把模型行为放进真实业务流程里：既要有流式体验，又要能暂停；既要能执行工具，又要能防误操作；既要保存上下文，又要支持重生成和历史恢复。

## 如果我要继续演进，我会怎么做

这部分很适合面试末尾讲，显得你有架构视野。

### 1）做权限体系更细化

- 不同用户可见不同工具；
- 不同动作走不同审批策略；
- 对网吧范围做更严格权限过滤。

### 2）做更强的可观测性

- 接 LangSmith 或自建 tracing；
- 做节点耗时、工具成功率、人工驳回率统计；
- 建立会话质量评估指标。

### 3）做知识增强

- 把业务 FAQ、运维手册、异常案例接进来；
- 做 FAQ/RAG 和工具调用融合；
- 对“问答类”问题和“执行类”问题走不同策略。

### 4）做多 Agent 或子任务拆解

- 查询型 Agent；
- 执行型 Agent；
- 报表分析型 Agent；
- 工单协同型 Agent。

### 5）做更强的前端工作台

- 展示执行链路图；
- 展示 tool artifact 卡片；
- 增加审批记录、操作审计、失败重试入口。

这部分可以收尾成一句：

> 我现在这个版本已经把业务 Agent 的主骨架搭起来了，下一步的重点不是继续堆 prompt，而是往权限、观测、知识和多 Agent 协作方向升级，让它从“可用”走向“可运营”。
